In Zed's `keymap.json`, a binding that looks duplicated between the `Workspace` context and the `Editor` context is often NOT redundant — it's there to defeat a Zed built-in Editor-context default that would otherwise shadow the Workspace-context binding.

Concrete example that bit us: `Ctrl+H` is bound to `workspace::ActivatePaneLeft` in the Workspace context, and was re-declared in the Editor context pointing at the same action. It looked redundant. It wasn't: Zed's default Editor-context `Ctrl+H` is `editor::ToggleReplace` (find-and-replace), and the more-specific Editor context wins, so without the Editor-context override, `Ctrl+H` inside an editor opens find/replace instead of navigating panes.

Task 26 (2026-04-10) removed the "redundant" Editor-context `ctrl-h`/`ctrl-l` and broke pane navigation inside editors. Fix was commit `a00d7b7` — restored the overrides and added an explanatory comment block.

**Why:** Zed context resolution is most-specific-first. Editor context beats Workspace context. Built-in defaults exist in specific contexts (Editor, Pane, Terminal, etc.) and will shadow your broader custom bindings unless you re-declare them in the same narrow context.

**How to apply:** Before removing an apparently-redundant binding from `keymap.json`, check whether the key has a Zed built-in default in the *narrower* context. If it does, the "redundancy" is a shadow-override and must be kept. Common culprits to check: `Ctrl+H` (editor::ToggleReplace), `Ctrl+F` (editor::FindInFile), `Ctrl+G` (editor::GoToLine), and anything in the `--- Search ---` / `--- Editing ---` groups of Zed's macOS default set. When in doubt, test the binding inside an editor pane before committing a "cleanup."
