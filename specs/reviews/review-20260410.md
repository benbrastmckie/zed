## High Priority Issues

### Outdated Claude model IDs in agent configuration
**Files**:
- `settings.json:39` -- `claude-sonnet-4-20250514`
- `settings.json:44` -- `claude-opus-4-20250514`
- `docs/general/settings.md:59,64` -- same IDs documented

**Description**: The Zed Agent Panel's `default_model` and `inline_alternatives` point to Claude 4 model IDs from May 2025. The current model family is Claude 4.5/4.6. Latest IDs: `claude-opus-4-6`, `claude-sonnet-4-6`, `claude-haiku-4-5-20251001`.

**Impact**: The Zed Agent Panel (Ctrl+?) uses a year-old Sonnet/Opus pair instead of the latest models, leaving capability on the table. Documentation in `docs/general/settings.md` propagates the stale IDs to readers setting up the editor.

**Recommended fix**: Update `settings.json` to:
```jsonc

"inline_alternatives": [
  { "provider": "anthropic", "model": "claude-opus-4-6" }
]
```
Then mirror the change in `docs/general/settings.md:59,64`.

## Medium Priority Issues

### Runtime TTS log files tracked in git
**Files**:
- `specs/tmp/claude-tts-last-notify`
- `specs/tmp/claude-tts-notify.log`

**Description**: These files are produced by a Claude Code notification hook and mutate on every run. Both are currently tracked by git (confirmed via `git ls-files specs/tmp/`), so every session dirties the working tree and produces noisy diffs.

**Impact**: Persistent `modified` status on every Claude Code invocation; noisy `git status`; unnecessary history churn.

**Recommended fix**: `git rm --cached specs/tmp/claude-tts-last-notify specs/tmp/claude-tts-notify.log` and add `specs/tmp/` (or at least those two files) to `.gitignore`.

### README / keymap.json platform-shortcut mismatch
**Files**:
- `README.md:21-30` -- uses `Cmd+P`, `Cmd+S`, `Cmd+\``, `Cmd+Shift+F`, `Cmd+Shift+P`
- `keymap.json:46-116` -- default-reference comment block lists the same shortcuts as `Ctrl+P`, `Ctrl+S`, `Ctrl+\``, etc.

**Description**: The repo declares macOS as its target platform, and the README correctly uses Cmd+ modifiers for default shortcuts. The quick-reference comment block in `keymap.json`, however, lists Ctrl+ modifiers throughout -- inherited from a pre-macOS-rewrite of the config. A user reading `keymap.json` to learn defaults sees incorrect bindings.

**Impact**: Documentation drift; confusing for new macOS users browsing `keymap.json`.

**Recommended fix**: Rewrite the default-reference comment block in `keymap.json` to use `Cmd+` for all default bindings except the ones that genuinely are `Ctrl+` on macOS (the four custom ones: `Ctrl+Shift+A`, `Ctrl+H`, `Ctrl+L`, `Ctrl+?`).
